# Scenario 1: Real-World Application Migration

**Priority:** 1 - KubernetesPlugin Validation  
**Goal:** Verify KubernetesPlugin correctly cleans all resource types for migration

## Objective

Deploy real-world applications and validate that:
1. All resource types are exported correctly
2. KubernetesPlugin generates appropriate cleanup patches
3. No manual intervention needed for metadata cleanup
4. Application starts successfully on target cluster

## Test Application: WordPress with MySQL

A comprehensive multi-tier application based on Kubernetes tutorial, significantly extended with:
- **MySQL 8.0** backend with persistent storage
- **WordPress 6** with PHP-FPM
- **NGINX** reverse proxy serving static files
- **Multi-container pod** architecture
- **Automated installation** via Job
- **Secret management** via Kustomize
- **Persistent storage** for both MySQL and WordPress data

**Source:** https://github.com/konveyor-ecosystem/kubectl-migrate/tree/main/sample-resources/wordpress

## Resources Involved

The application tests these resource types:
- **Deployments** (wordpress-mysql, wordpress with multi-container)
- **Services** (ClusterIP for MySQL, LoadBalancer for WordPress)
- **PersistentVolumeClaims** (mysql-pv-claim, wordpress-pv-claim)
- **ConfigMap** (nginx-config with NGINX configuration)
- **Secret** (wordpress-secrets generated by Kustomize)
- **Job** (wordpress-install for automated setup)

## Setup - Deploy WordPress on Source Cluster

### 1. Download WordPress Sample Resources

```bash
# Create working directory
mkdir -p ~/crane-test-wordpress
cd ~/crane-test-wordpress

# Download WordPress sample resources
curl -LO https://github.com/konveyor-ecosystem/kubectl-migrate/archive/refs/heads/main.tar.gz
tar xzf main.tar.gz
cd kubectl-migrate-main/sample-resources/wordpress/

# Verify files
ls -la
# Should see: deploy.sh, destroy.sh, mysql-deployment.yaml, wordpress-deployment.yaml, 
#             nginx-config.yaml, wordpress-install-job.yaml, kustomization.yaml
```

### 2. Deploy to Source Cluster

```bash
# Switch to source cluster
kubectl config use-context <source-context>

# Deploy using provided script
./deploy.sh

# The script will:
# 1. Generate random passwords in .env file
# 2. Apply all resources using Kustomize
# 3. Wait for deployments to be ready
# 4. Wait for WordPress installation job to complete
```

**Expected output:**
```
# Using kubectl context: <your-source-context>
persistentvolumeclaim/mysql-pv-claim created
persistentvolumeclaim/wordpress-pv-claim created
service/wordpress-mysql created
service/wordpress created

deployment.apps/wordpress created

job.batch/wordpress-install created
secret/wordpress-secrets created
Waiting for mysql, wordpress and install job to be completed...

deployment.apps/wordpress condition met
job.batch/wordpress-install condition met
```

### 3. Validate Application Running

```bash
# Check all resources
kubectl get all,pvc,cm,secret -l app=wordpress
```

## Migration with Crane

### Step 1: Export

```bash
# Create migration working directory
cd ~/crane-test-wordpress
mkdir migration
cd migration

# Export the default namespace (or whichever namespace you used)
crane export

# Inspect exported resources
ls -la export/resources/default/

# Count exported files
ls export/resources/default/ | wc -l
```

**Validation checklist:**
- [ ] Both Deployments exported (wordpress, wordpress-mysql)
- [ ] Both Services exported
- [ ] Both PVCs exported
- [ ] ConfigMap exported (nginx-config)
- [ ] Secret exported (wordpress-secrets)
- [ ] Job exported (wordpress-install)
- [ ] ReplicaSets exported (auto-generated)
- [ ] Pods NOT exported (runtime resources, correct behavior)

**Common issues to watch:**
- Auto-generated ServiceAccount tokens might be exported
- Check if all expected resource types are present

### Step 2: Inspect Exported Resources

Examine different resource types:

```bash
# Deployment (multi-container)
cat export/resources/default/Deployment_apps_v1_default_wordpress.yaml | head -50

# Check for runtime metadata to be cleaned:
grep -E "(uid|resourceVersion|creationTimestamp|generation|managedFields|status)" \
  export/resources/default/Deployment_apps_v1_default_wordpress.yaml | wc -l
# Should show many matches (these need to be cleaned)

# PersistentVolumeClaim
cat export/resources/default/PersistentVolumeClaim_default_mysql-pv-claim.yaml

# Job
cat export/resources/default/Job_batch_v1_default_wordpress-install.yaml

# ConfigMap with embedded NGINX config
cat export/resources/default/ConfigMap_default_nginx-config.yaml
```

**Document:**
- What server-managed fields are present?
- Any unexpected fields?
- Any resource-specific metadata?
- How does multi-container Deployment look?

### Step 3: Transform with KubernetesPlugin

```bash
# Run transform
crane transform

# Inspect output
tree transform/10_KubernetesPlugin/

# Check patches directory
ls -la transform/10_KubernetesPlugin/patches/

# Count patches (should match resource count)
echo "Resources: $(ls export/resources/default/ | wc -l)"
echo "Patches: $(ls transform/10_KubernetesPlugin/patches/ | wc -l)"
```

**Validation checklist:**
- [ ] One patch file per exported resource
- [ ] Patches directory matches resource count
- [ ] kustomization.yaml references all resources and patches

### Step 4: Inspect Patches for Different Resource Types

```bash
# Deployment patch
echo "=== Deployment patch ==="
cat transform/10_KubernetesPlugin/patches/default--apps-v1--Deployment--wordpress.patch.yaml

# PVC patch
echo "=== PVC patch ==="
cat transform/10_KubernetesPlugin/patches/default--v1--PersistentVolumeClaim--mysql-pv-claim.patch.yaml

# Job patch
echo "=== Job patch ==="
cat transform/10_KubernetesPlugin/patches/default--batch-v1--Job--wordpress-install.patch.yaml

# ConfigMap patch
echo "=== ConfigMap patch ==="
cat transform/10_KubernetesPlugin/patches/default--v1--ConfigMap--nginx-config.patch.yaml

# Secret patch
echo "=== Secret patch ==="
cat transform/10_KubernetesPlugin/patches/default--v1--Secret--wordpress-secrets.patch.yaml
```

**Validation for each resource type:**
- [ ] Removes `metadata.uid`
- [ ] Removes `metadata.resourceVersion`
- [ ] Removes `metadata.creationTimestamp`
- [ ] Removes `metadata.generation` (if present)
- [ ] Removes `metadata.managedFields`
- [ ] Removes `status` section
- [ ] Preserves user-defined fields (labels, annotations, spec)
- [ ] Preserves multi-container configuration (wordpress deployment)
- [ ] Preserves ConfigMap data (nginx config)
- [ ] Preserves PVC storage requests

**Document any incorrect or missing patches.**

### Step 5: Validate Multi-Container Pod Configuration

```bash
# Check that multi-container deployment is preserved correctly
kubectl kustomize transform/10_KubernetesPlugin/ | \
  yq eval 'select(.kind == "Deployment" and .metadata.name == "wordpress") | .spec.template.spec.containers | length'
# Should output: 2 (wordpress-fpm + nginx containers)

# Check container names preserved
kubectl kustomize transform/10_KubernetesPlugin/ | \
  yq eval 'select(.kind == "Deployment" and .metadata.name == "wordpress") | .spec.template.spec.containers[].name'
# Should output:
# wordpress
# nginx
```

### Step 6: Preview Cleaned Resources

```bash
# Preview all cleaned resources
kubectl kustomize transform/10_KubernetesPlugin/ > cleaned-preview.yaml

# Check that server-managed fields are removed
grep -E "(uid:|resourceVersion:|creationTimestamp:|managedFields:|^status:)" cleaned-preview.yaml
# Should return nothing or minimal matches

# Verify important fields preserved
echo "=== User-defined preserved ==="
grep "app: wordpress" cleaned-preview.yaml | wc -l  # Should show labels
grep "tier: " cleaned-preview.yaml | wc -l          # Should show tier labels
grep "storage: 1Gi" cleaned-preview.yaml | wc -l    # Should show PVC storage
```

### Step 7: Generate Final Output

```bash
# Run crane apply
crane apply

# Inspect output
cat output/output.yaml | head -100

# Verify resource count
grep "^kind:" output/output.yaml | sort | uniq -c

# Expected counts:
# 1 ConfigMap
# 2 Deployment
# 1 Job
# 2 PersistentVolumeClaim
# 2 Service
# 1 Secret
# (+ possibly ReplicaSets)
```

### Step 8: Validate Output

```bash
# Syntax validation
kubectl apply --dry-run=client -f output/output.yaml

# Check for any errors or warnings
```

**Document any validation failures.**

### Step 9: Deploy to Target Cluster

```bash
# Switch to target cluster
kubectl config use-context <target-context>

# Create namespace if different
# kubectl create namespace wordpress-migrated

# Apply migrated resources
kubectl apply -f output/output.yaml

# Watch deployment
kubectl get pods -w
```

**Validation:**
- [ ] All resources created successfully
- [ ] No errors during apply
- [ ] Pods start successfully
- [ ] Services are created
- [ ] PVCs are bound (storage must be available)
- [ ] Job runs and completes

**Common issues:**
- Storage class might not exist on target cluster
- LoadBalancer type might not be supported (use port-forward instead)
- Different default storage class on target

### Step 10: Functional Testing

```bash
# Wait for deployments to be ready
kubectl wait --for=condition=available --timeout=300s deployment/wordpress-mysql
kubectl wait --for=condition=available --timeout=300s deployment/wordpress

# Wait for installation job
kubectl wait --for=condition=complete --timeout=300s job.batch/wordpress-install

# Test WordPress access
kubectl port-forward svc/wordpress 8080:80 &
curl -I http://localhost:8080
# Should return: HTTP/1.1 200 OK

# Test actual content
curl http://localhost:8080 | grep -i wordpress
# Should find WordPress content

# Check multi-container pod
kubectl get pod -l tier=frontend -o jsonpath='{.items[0].spec.containers[*].name}'
# Should show: wordpress nginx

# Verify MySQL connectivity from WordPress
kubectl exec deployment/wordpress -c wordpress -- \
  sh -c 'mysqladmin ping -h wordpress-mysql -u wordpress -p$WORDPRESS_DB_PASSWORD'
# Should return: mysqld is alive

# Check Job logs to see installation was successful
kubectl logs job/wordpress-install
# Should show WordPress installation steps
```

## Common Issues to Document

1. **Resource Types Not Exported**
   - Which types?
   - Why not exported?
   - Should they be?

2. **Incorrect or Missing Patches**
   - Which resource type?
   - What fields not cleaned?
   - What fields incorrectly removed?

3. **Multi-Container Pod Issues**
   - Were both containers preserved?
   - Shared volume configuration correct?
   - Container order preserved?

4. **Apply Failures**
   - Error messages
   - Which resources failed?
   - Why (API version, field validation, etc.)?

5. **Storage Issues**
   - StorageClass not available
   - PVC binding fails
   - Different storage defaults on target

6. **Job Execution Issues**
   - Job fails to complete
   - WordPress installation errors
   - Secret access issues

## Expected Results

✅ **Success criteria:**
- All resource types export cleanly
- KubernetesPlugin generates correct patches for all types
- Multi-container Deployment handled correctly
- ConfigMap with embedded config preserved
- Job resource type supported
- Application deploys and runs on target cluster without manual intervention

⚠️ **Acceptable issues:**
- LoadBalancer service type not supported (use port-forward)
- StorageClass name differs (manual adjustment needed)
- Auto-generated resources (ServiceAccount tokens) need cleanup

❌ **Blocking issues:**
- Resource types not exported
- Patches missing for resource types
- Incorrect patches breaking resources
- Multi-container configuration lost
- ConfigMap data corrupted
- Application fails to start due to metadata issues
