# Scenario 1: Real-World Application Migration

**Priority:** 1 - KubernetesPlugin Validation  
**Goal:** Verify KubernetesPlugin correctly cleans all resource types for migration

## Objective

Deploy real-world applications and validate that:
1. All resource types are exported correctly
2. KubernetesPlugin generates appropriate cleanup patches
3. No manual intervention needed for metadata cleanup
4. Application starts successfully on target cluster

## Test Application: WordPress with MySQL

A comprehensive multi-tier application based on Kubernetes tutorial, significantly extended with:
- **MySQL 8.0** backend with persistent storage
- **WordPress 6** with PHP-FPM
- **NGINX** reverse proxy serving static files
- **Multi-container pod** architecture
- **Automated installation** via Job
- **Secret management** via Kustomize
- **Persistent storage** for both MySQL and WordPress data

**Application files:** [sample-apps/wordpress/](./sample-apps/wordpress/)  
**Source:** https://github.com/konveyor-ecosystem/kubectl-migrate/tree/main/sample-resources/wordpress

## Resources Involved

The application tests these resource types:
- **Deployments** (wordpress-mysql, wordpress with multi-container)
- **Services** (ClusterIP for MySQL, LoadBalancer for WordPress)
- **PersistentVolumeClaims** (mysql-pv-claim, wordpress-pv-claim)
- **ConfigMap** (nginx-config with NGINX configuration)
- **Secret** (wordpress-secrets generated by Kustomize)
- **Job** (wordpress-install for automated setup)

## Migration with Crane

### Step 1: Create Migration Workspace

```bash
# Create migration working directory (outside wordpress directory)
cd /path/to/test-day-june2026
mkdir -p migration-wordpress
cd migration-wordpress

# Note: We're now working outside the sample-apps directory
# The WordPress app is running on the source cluster
```

### Step 2: Export

```bash
# Export the default namespace (or whichever namespace you used)
crane export

# Inspect exported resources
ls -la export/resources/default/

# Count exported files
echo "Total exported files: $(ls export/resources/default/ | wc -l)"

# List resource types
ls export/resources/default/ | cut -d'_' -f1 | sort -u
```

**Validation checklist:**
- [ ] Both Deployments exported (wordpress, wordpress-mysql)
- [ ] Both Services exported (wordpress, wordpress-mysql)
- [ ] Both PVCs exported (mysql-pv-claim, wordpress-pv-claim)
- [ ] ConfigMap exported (nginx-config)
- [ ] Secret exported (wordpress-secrets)
- [ ] Job exported (wordpress-install)
- [ ] ReplicaSets exported (auto-generated by Deployments)

**Common observations:**
- Auto-generated ServiceAccount tokens might be exported
- ReplicaSets are exported (owned by Deployments)
- Pods are NOT exported (runtime resources, correct behavior)

### Step 3: Inspect Exported Resources

Examine different resource types for metadata that needs cleanup:

```bash
# Multi-container Deployment
cat export/resources/default/Deployment_apps_v1_default_wordpress.yaml | head -60

# Check for runtime metadata
grep -E "(uid|resourceVersion|creationTimestamp|generation|managedFields)" \
  export/resources/default/Deployment_apps_v1_default_wordpress.yaml | wc -l
# Should show many matches - these need to be cleaned

# PersistentVolumeClaim
cat export/resources/default/PersistentVolumeClaim_default_mysql-pv-claim.yaml

# Job
cat export/resources/default/Job_batch_v1_default_wordpress-install.yaml

# ConfigMap with embedded NGINX config
cat export/resources/default/ConfigMap_default_nginx-config.yaml | less
```

**Document:**
- What server-managed fields are present?
- Are there any unexpected fields?
- How does the multi-container Deployment look?
- Is the ConfigMap data intact?

### Step 4: Transform with KubernetesPlugin

```bash
# Run transform
crane transform

# Inspect transform output
tree transform/10_KubernetesPlugin/

# Verify patches were generated
echo "Exported resources: $(ls export/resources/default/ | wc -l)"
echo "Generated patches: $(ls transform/10_KubernetesPlugin/patches/ | wc -l)"
# Counts should match (one patch per resource)
```

**Validation checklist:**
- [ ] Transform stage created: `10_KubernetesPlugin/`
- [ ] Patches directory contains patch files
- [ ] Resources directory contains YAML files
- [ ] kustomization.yaml exists and references resources and patches

### Step 5: Inspect Patches for Different Resource Types

```bash
# Deployment patch (multi-container)
echo "=== Deployment wordpress patch ==="
cat transform/10_KubernetesPlugin/patches/default--apps-v1--Deployment--wordpress.patch.yaml

# Deployment patch (mysql)
echo "=== Deployment wordpress-mysql patch ==="
cat transform/10_KubernetesPlugin/patches/default--apps-v1--Deployment--wordpress-mysql.patch.yaml

# PVC patches
echo "=== PVC mysql patch ==="
cat transform/10_KubernetesPlugin/patches/default--v1--PersistentVolumeClaim--mysql-pv-claim.patch.yaml

echo "=== PVC wordpress patch ==="
cat transform/10_KubernetesPlugin/patches/default--v1--PersistentVolumeClaim--wordpress-pv-claim.patch.yaml

# Job patch
echo "=== Job patch ==="
cat transform/10_KubernetesPlugin/patches/default--batch-v1--Job--wordpress-install.patch.yaml

# ConfigMap patch
echo "=== ConfigMap patch ==="
cat transform/10_KubernetesPlugin/patches/default--v1--ConfigMap--nginx-config.patch.yaml

# Secret patch
echo "=== Secret patch ==="
cat transform/10_KubernetesPlugin/patches/default--v1--Secret--wordpress-secrets.patch.yaml
```

**Validation for each resource type - patches should:**
- [ ] Remove `metadata.uid`
- [ ] Remove `metadata.resourceVersion`
- [ ] Remove `metadata.creationTimestamp`
- [ ] Remove `metadata.generation` (if present)
- [ ] Remove `metadata.managedFields`
- [ ] Remove `status` section
- [ ] Preserve user-defined labels and annotations
- [ ] Preserve all spec fields
- [ ] Preserve ConfigMap data integrity
- [ ] Preserve Secret data
- [ ] Preserve multi-container configuration

**Document any incorrect or missing patches.**

### Step 6: Validate Multi-Container Pod Configuration

The wordpress Deployment uses a multi-container pod (wordpress-fpm + nginx). Verify this is preserved:

```bash
# Check container count
kubectl kustomize transform/10_KubernetesPlugin/ | \
  yq eval 'select(.kind == "Deployment" and .metadata.name == "wordpress") | .spec.template.spec.containers | length'
# Should output: 2

# Check container names
kubectl kustomize transform/10_KubernetesPlugin/ | \
  yq eval 'select(.kind == "Deployment" and .metadata.name == "wordpress") | .spec.template.spec.containers[].name'
# Should output:
# wordpress
# nginx

# Verify shared volume mount
kubectl kustomize transform/10_KubernetesPlugin/ | \
  yq eval 'select(.kind == "Deployment" and .metadata.name == "wordpress") | .spec.template.spec.volumes[] | select(.name == "wordpress-persistent-storage")'
# Should show PVC reference
```

### Step 7: Validate ConfigMap Data Integrity

NGINX configuration in ConfigMap should be preserved correctly:

```bash
# Extract ConfigMap data
kubectl kustomize transform/10_KubernetesPlugin/ | \
  yq eval 'select(.kind == "ConfigMap" and .metadata.name == "nginx-config") | .data."nginx.conf"' | head -20

# Verify key configuration lines are present
kubectl kustomize transform/10_KubernetesPlugin/ | \
  yq eval 'select(.kind == "ConfigMap" and .metadata.name == "nginx-config") | .data."nginx.conf"' | \
  grep -E "(fastcgi_pass|location ~ \.php|try_files)"
# Should find FastCGI and PHP handling configuration
```

### Step 8: Preview Cleaned Resources

```bash
# Generate preview of all cleaned resources
kubectl kustomize transform/10_KubernetesPlugin/ > cleaned-preview.yaml

# Verify server-managed fields removed
grep -E "(uid:|resourceVersion:|creationTimestamp:|managedFields:|^status:)" cleaned-preview.yaml
# Should return nothing or minimal matches

# Verify important fields preserved
echo "=== Preserved labels ==="
grep "app: wordpress" cleaned-preview.yaml | wc -l  
grep "tier: " cleaned-preview.yaml | wc -l

echo "=== Preserved storage ==="
grep "storage: 1Gi" cleaned-preview.yaml

echo "=== Preserved multi-container ==="
grep -A 2 "containers:" cleaned-preview.yaml | grep "name: wordpress"
grep -A 2 "containers:" cleaned-preview.yaml | grep "name: nginx"
```

### Step 9: Generate Final Output

```bash
# Run crane apply
crane apply

# Inspect output
less output/output.yaml

# Count resources by type
grep "^kind:" output/output.yaml | sort | uniq -c

# Expected counts:
# 1 ConfigMap
# 2 Deployment
# 1 Job
# 2 PersistentVolumeClaim
# 2 Service
# 1 Secret
# (+ possibly 2 ReplicaSets)
```

### Step 10: Validate Output

```bash
# Syntax validation
kubectl apply --dry-run=client -f output/output.yaml

# Should complete without errors
```

**Document any validation failures.**

### Step 11: Deploy to Target Cluster

```bash
# Switch to target cluster
kubectl config use-context <target-context>

# Create namespace if using different namespace
# kubectl create namespace wordpress-prod

# Apply migrated resources
kubectl apply -f output/output.yaml

# Watch deployment progress
kubectl get pods -w
# Press Ctrl+C when pods are running
```

**Validation:**
- [ ] All resources created successfully
- [ ] No errors during apply
- [ ] MySQL pod starts (1/1 containers)
- [ ] WordPress pod starts (2/2 containers - wordpress + nginx)
- [ ] Services created
- [ ] PVCs bound to storage
- [ ] Job runs and completes
- [ ] No warnings or errors in pod logs

**Common issues:**
- StorageClass might not exist or differ on target cluster
- LoadBalancer Service type might not be supported (workaround: port-forward)
- Job might take time to complete (waits for MySQL readiness)

### Step 12: Functional Testing on Target

```bash
# Wait for all resources to be ready
kubectl wait --for=condition=available --timeout=300s deployment/wordpress-mysql
kubectl wait --for=condition=available --timeout=300s deployment/wordpress

# Wait for WordPress installation
kubectl wait --for=condition=complete --timeout=300s job.batch/wordpress-install

# Test WordPress accessibility
kubectl port-forward svc/wordpress 8080:80 &
sleep 2
curl -I http://localhost:8080
# Should return: HTTP/1.1 200 OK

# Test actual WordPress content
curl -s http://localhost:8080 | grep -i wordpress
# Should find WordPress-specific content

# Verify multi-container pod
kubectl get pod -l tier=frontend -o jsonpath='{.items[0].spec.containers[*].name}'
# Should output: wordpress nginx

# Verify MySQL connectivity from WordPress pod
kubectl exec deployment/wordpress -c wordpress -- \
  sh -c 'mysqladmin ping -h wordpress-mysql -u wordpress -p$WORDPRESS_DB_PASSWORD 2>/dev/null'
# Should output: mysqld is alive

# Check installation job logs
kubectl logs job/wordpress-install | tail -20
# Should show successful WordPress installation

# Cleanup port-forward
pkill -f "port-forward.*wordpress"
```
